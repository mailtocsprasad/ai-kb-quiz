# Epic 4: Programmable Tool Calling

Programmable Tool Calling is a runtime pattern where the premium model generates a Python
extraction script on-the-fly. The script is validated via RestrictedPython AST analysis,
then executed in a subprocess wrapped in a Windows Job Object with CPU/memory limits and
a kill-on-timeout policy. Only the compact result is returned to the model.

## Story 4.1 — Premium model generates and runs extraction scripts

```gherkin
Feature: Programmable Tool Calling
  As the quiz engine
  I want the premium model to generate targeted extraction scripts at runtime
  So that it only processes the data it needs and token usage is controlled

  Scenario: Model generates script and receives compact result
    Given PTC has compressed KB chunks to 750 tokens
    And the task is "generate a conceptual question about kernel pool allocators"
    When Programmable Tool Calling runs
    Then the premium model generates a Python extraction script targeting the compact input
    And the script passes RestrictedPython AST validation
    And the script runs in a subprocess wrapped in a Windows Job Object
    And only the script output (under 500 tokens) is returned to the model
    And the model uses that output to generate the question

  Scenario: Generated script execution fails
    Given a Python extraction script is generated by the premium model
    When the subprocess raises an exception
    Then the error and the failed script are logged
    And the pipeline falls back to passing the PTC-compressed output directly
    And the premium model still never receives raw KB chunks

  Scenario: Sandbox timeout exceeded
    Given a generated script runs longer than prog_tool_calling.sandbox_timeout_sec
    When the Job Object kill-on-close policy fires
    Then the subprocess is terminated
    And the fallback path is used
    And the timeout event is logged with the script content

  Scenario: Programmable Tool Calling stats logged
    Given a Programmable Tool Calling run completes
    Then the session log records:
      | field                   | example value |
      | ptc_compressed_tokens   | 750           |
      | prog_tool_output_tokens | 420           |
      | script_generated        | true          |
      | fallback_used           | false         |
```

## Story 4.2 — RestrictedPython AST validation

```gherkin
Feature: Script safety validation
  As the quiz engine
  I want model-generated scripts validated before execution
  So that dangerous operations are blocked before they run

  Scenario: Safe script passes validation
    Given a generated script imports only builtins (str, list, dict operations)
    When RestrictedPython validates the AST
    Then validation passes
    And the script proceeds to subprocess execution

  Scenario: Script with dangerous import blocked
    Given a generated script contains "import os" or "import subprocess"
    When RestrictedPython validates the AST
    Then validation fails immediately
    And the script is not executed
    And the violation is logged with the script content
    And the fallback path is used

  Scenario: Script with file I/O blocked
    Given a generated script contains open() or file write operations
    When RestrictedPython validates the AST
    Then validation fails
    And the fallback path is used

  Scenario: Script with network access blocked
    Given a generated script contains socket or urllib usage
    When RestrictedPython validates the AST
    Then validation fails
    And the fallback path is used
```

## Story 4.3 — Windows Job Object isolation

```gherkin
Feature: Job Object process isolation
  As the quiz engine
  I want subprocess execution wrapped in a Windows Job Object
  So that runaway or malicious scripts cannot escape resource limits

  Scenario: Job Object enforces memory limit
    Given a generated script allocates more memory than the configured limit
    When the Job Object memory quota is exceeded
    Then the subprocess is terminated
    And MemoryError is logged
    And the fallback path is used

  Scenario: Job Object kills child processes
    Given a generated script attempts to spawn child processes
    When the Job Object limit fires
    Then spawned children are also terminated
    And the event is logged

  Scenario: Job Object cleans up on engine exit
    Given a quiz session ends normally or via exception
    When the engine process closes
    Then all Job Object-wrapped subprocesses are automatically terminated
    And no orphan processes remain
```
